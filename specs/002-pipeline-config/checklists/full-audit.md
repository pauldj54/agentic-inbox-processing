- [x] CHK008 - Is "classification step" unambiguously defined — does it include only Step 3 (LLM classification), or also Steps 4a (dedup) and 4b (content storage)? [Clarity, Spec §FR-004 — says "classification step" but plan shows Steps 3–5 skipped] — ✅ Fixed: FR-004 now clarifies "The classification step (encompassing LLM classification, deduplication, and PE-event creation — implementation Steps 3–5) MUST be skipped entirely."

Two different metrics for two different routing decisions in two different modes.

## Scenario Coverage

- [x] CHK021 - Are requirements defined for concurrent mode changes — what if the env var is changed while the agent process is running (hot reload vs restart-only)? [Coverage, Spec §Edge Cases — mentions "restart" but FR-006 says "read at startup" without explicitly ruling out hot reload] — ✅ Explicitly addressed: Edge Case 2 says "The new mode applies only after agent restart." FR-006 says "MUST be read at agent startup. Changes MUST take effect on the next agent restart." Hot reload is ruled out.
- [x] CHK022 - Are requirements defined for what the triage-complete message includes when an email has zero attachments? [Coverage, Gap — schema shows `attachmentPaths` array but no zero-attachment example] — ✅ Fixed: New Edge Case added: "email has zero attachments → `hasAttachments: false`, `attachmentsCount: 0`, `attachmentPaths: []`. Pre-processing is a no-op and the email proceeds to routing."
- [x] CHK023 - Are requirements defined for what happens when the triage-complete queue does not exist on the target namespace? [Coverage, Gap — external namespace may not have the queue pre-created] — ✅ Fixed: New Edge Case added specifying the SDK error is treated as an unrecoverable routing failure → DLQ on primary namespace. New assumption added that queues must be pre-created.
- [x] CHK024 - Are requirements defined for partial pre-processing failure in triage-only mode (e.g., 2 of 3 attachments fail OCR)? [Coverage, Gap — Edge Cases only cover "all attachment processing fails"] — ✅ Fixed: New Edge Case added: partial failures allow email to continue processing with successfully extracted attachments. Only ALL attachments failing triggers DLQ.
- [x] CHK025 - Are requirements defined for observability/metrics — how does an operator know how many emails were processed in each mode over time? [Coverage, Gap — FR-013 covers dashboard display but not aggregate metrics] — ✅ Out of scope for this feature. FR-008 + CAR-008 mandate structured logging of pipeline mode per email. Aggregate metrics are derived from log analytics queries on these structured logs — an operational tooling concern, not a feature requirement.

## Edge Case Coverage

- [x] CHK026 - Is the fallback behavior when the external SB namespace is unreachable specified with sufficient detail — does the email stay in the intake queue (retry) or go to DLQ (fail-fast)? [Edge Case, Spec §Edge Cases item 6 — says "dead-letter queue" but contracts say DLQ on primary namespace] — ✅ Edge Case 6 says "route to the dead-letter queue on the primary namespace." Contracts §3 confirms DLQ on primary. Research §6 confirms fail-fast design.
- [x] CHK027 - Are retry semantics defined for the external namespace connection (how many retries, backoff, timeout)? [Edge Case, Gap] — ✅ Azure ServiceBus SDK has built-in retry policy (3 retries with exponential backoff by default). Additional retry logic is explicitly not added — research §6 confirms fail-fast-to-DLQ design. SDK default retries are sufficient.
- [x] CHK028 - Are requirements defined for the case where `PIPELINE_MODE` is set to `"triage-only"` but `TRIAGE_COMPLETE_QUEUE` points to a queue that already has consumers expecting classified messages? [Edge Case, Gap — misconfiguration scenario] — ✅ Operational misconfiguration risk, not a system requirement. The contracts §1 message schema documents the exact triage-complete format; consumer compatibility is the consumer team's responsibility. The producer sends the documented schema.
- [x] CHK029 - Is the behavior specified when `TRIAGE_COMPLETE_SB_NAMESPACE` is set but `PIPELINE_MODE` is `"full"`? [Edge Case, Gap — external namespace config is only relevant in triage-only mode; is it silently ignored?] — ✅ Fixed: New assumption added to spec: "When `TRIAGE_COMPLETE_SB_NAMESPACE` is set but `PIPELINE_MODE` is `\"full\"`, the external namespace configuration is silently ignored. The external Service Bus client is only instantiated in triage-only mode."
